# /notify:setup

Configure the `notify` plugin end-to-end: Pushover tokens, `~/.claude-workbench/.env`, and the CLI shim. **No shell rc edits required** — the dispatcher auto-loads tokens from `~/.claude-workbench/.env`.

## 1. Pre-flight

- Env file path: `~/.claude-workbench/.env`.
- Config path: `~/.claude-workbench/notify-config.json`.
- If config already exists and `--reset` is NOT set, treat this as update. Skip token capture unless the user asks to reconfigure.
- Create `~/.claude-workbench/` if missing: `mkdir -p ~/.claude-workbench`.

## 2. Confirm provider

`AskUserQuestion` once — v0.1.0 only supports Pushover, this is a confirmation:

> Configure Pushover now? (Other providers — ntfy / Slack / Telegram — land in v0.2+.)
>
> - Yes, I have Pushover keys.
> - Yes, but I don't have keys yet — show me where to get them.
> - Skip for now (just install the CLI shim).

If the user picks "show me where": point at `https://pushover.net`. User key is on the dashboard (30-char, starts with `u`); app token comes from creating an Application in the dashboard (30-char, starts with `a`). Have them return to this command.

## 3. Capture tokens (if user proceeds)

Use `AskUserQuestion` twice, separately (never log the actual token values back to the user):

> Pushover user key (starts with `u`)?

> Pushover app token (starts with `a`)?

Sanity-check each:
- Length ≥ 20.
- No whitespace, no quotes, no `=` character.

If either fails the check, ask again once. If still invalid, stop and tell the user to re-run `/notify:setup --reset`.

## 4. Write `.env` — this is the new store-tokens step

**DO NOT** tell the user to `export` anything in their shell rc. Instead, write the tokens to `~/.claude-workbench/.env` yourself:

```bash
umask 077
cat > ~/.claude-workbench/.env <<EOF
# Generated by /notify:setup at <now>
# This file is auto-loaded by notify-dispatch.py. Never commit it anywhere.
PUSHOVER_USER_KEY=<captured-user-key>
PUSHOVER_APP_TOKEN=<captured-app-token>
EOF
chmod 600 ~/.claude-workbench/.env
```

Use `Write` tool, not `Bash(cat:*)` with a heredoc containing the live tokens — keeping captured secrets out of the Bash-tool audit trail is safer. Construct the file contents in memory and `Write` in one shot.

If `.env` already exists and the user ran `--reset`: overwrite. If not `--reset`, merge — keep unrelated keys (e.g. `NTFY_TOPIC`) and only replace `PUSHOVER_*`.

## 5. Write / update notify-config.json

1. If config doesn't exist, copy `${CLAUDE_PLUGIN_ROOT}/templates/notify-config.example.json` → `~/.claude-workbench/notify-config.json`. The template already uses `${PUSHOVER_USER_KEY}` / `${PUSHOVER_APP_TOKEN}` references — no token values appear inside.
2. If it exists, leave it alone (the user may have tuned rules).

**Never** edit the config to inline literal token values. The whole point of this setup is that the only place tokens live is `~/.claude-workbench/.env`.

## 6. Install the CLI shim

Run `bash ${CLAUDE_PLUGIN_ROOT}/scripts/install-cli.sh` and surface its stdout verbatim. It:
- creates `~/.claude-workbench/bin/`
- symlinks `workbench-notify` into it
- prints PATH instructions

## 7. Emit next-steps block

Print (paraphrase the token values — never show the captured ones):

```
✓ notify configured.

Tokens live at:
  ~/.claude-workbench/.env    (chmod 600, auto-loaded by the dispatcher)

Config (with ${VAR} references, no literal tokens):
  ~/.claude-workbench/notify-config.json

To use the CLI from your shell (optional — only needed for manual testing):
  export PATH="$HOME/.claude-workbench/bin:$PATH"
Add that to ~/.bashrc / ~/.zshrc. On Windows, set it in the user Environment
Variables GUI. Slash commands (/notify:test) don't require this — they
prepend PATH themselves.

Verify:
  /notify:test
```

## Absolute rules

- **Never** echo captured tokens back to the user in plaintext after capture.
- **Never** write tokens inline into `notify-config.json`. Only `${VAR}` references.
- **Never** instruct the user to add `export PUSHOVER_*` to their shell rc. The `.env` file is the store; that's the whole point of this setup.
- **Always** chmod 600 the `.env` file on Unix-like systems (on Windows this is a no-op but doesn't hurt).
- **Always** run install-cli.sh at the end — siblings rely on the CLI being on PATH (via the shell OR via the auto-prepend that the hook scripts do for themselves).
- **Never** mark the task complete if `workbench-notify --health` returns non-zero after setup. Tell the user what to fix.
